# AI Rumor Town Design

## 项目定位

项目暂定名为《谣镇档案：怪谈调查局》。它是一个规则怪谈式 AI 推理游戏，玩家以调查员身份进入不同小镇案件，在固定真相的基础上，通过 NPC 对话、地点调查、规则验证和线索档案整理，最终提交真相报告。

第一版是前后端完整项目：

- 后端使用 Spring Boot。
- 前端使用 Vue 3、Vite、Pinia、Vue Router。
- 数据库使用 MySQL。
- AI 使用硅基流动 DeepSeek V4 Pro。
- 第一版使用游客玩家模式，同时保留用户表，方便后续升级注册登录。
- 第一版包含 3 个固定案件。
- 案件真相固定，AI 不能修改真相、凶手、关键证据或阶段规则。

项目核心体验是“固定推理骨架 + 规则怪谈氛围”。每个案件包含 2-3 条怪谈规则。玩家需要判断规则是迷信、误导、真实危险，还是关键线索的伪装。

## 首发案件

### 《三点十七分的钟声》

核心元素：钟楼、镇长失踪、红手套、旧矿难。

怪谈规则示例：

- 听见第十三下钟声后，不要回头看钟楼。
- 红手套不能被带进教堂。
- 如果有人说“钟早就修好了”，立刻结束对话。

真实推理内核：钟声是人为启动的机械信号，红手套关联旧矿难遇难者家属。

### 《玫瑰旅馆的第八间房》

核心元素：旅馆、密室、账本、第八把钥匙、双重身份。

怪谈规则示例：

- 旅馆只有七间房，如果前台给你第八把钥匙，不要接。
- 晚上听到隔壁敲三下，不要回应第四下。
- 不要相信镜子里看到的房号。

真实推理内核：第八间房是伪造身份、隐藏账本和转移证人的机关。

### 《雾港最后一班船》

核心元素：港口、无人渡船、船票名单、失踪人口、假身份。

怪谈规则示例：

- 雾起后不要数船票。
- 没有乘客的船不能登。
- 如果船员问你“今天是几号”，不要回答真实日期。

真实推理内核：空船、船票名单和假日期指向走私、失踪人口和身份替换。

## 核心玩法

玩家流程：

1. 进入案件档案馆。
2. 选择一个案件。
3. 阅读开场事件和 2-3 条怪谈规则。
4. 进入调查主界面。
5. 访问地点，和 NPC 对话。
6. 收集线索并查看已收集线索档案。
7. 随着关键线索出现，推进案件阶段。
8. 填写真相报告。
9. 获得 AI 评分复盘和结局。

案件阶段采用阶段式推进：

- 阶段 1：初访。开放基础地点和 NPC。
- 阶段 2：矛盾浮现。玩家收集到一定线索后，开放新追问和新地点。
- 阶段 3：旧案或核心秘密。解锁关键物证或隐藏 NPC。
- 阶段 4：提交真相报告。玩家整理结论并评分。

阶段推进由后端判断，依据包括关键线索、线索数量、已访问地点、对话次数和特定触发条件。

## 页面结构

### 案件档案馆页

展示 3 个案件档案卡。每个案件展示标题、封面、怪谈规则数量、预计调查时间、难度、当前进度和是否已通关。

### 案件详情页

展示案件开场事件、怪谈规则、已知地点、初始任务和“开始调查”按钮。规则不标注真假，只作为悬疑入口。

### 调查主界面

桌面端采用三栏布局：

- 左侧：案件地图或地点列表。
- 中间：当前地点描述、NPC 列表和对话窗口。
- 右侧：已收集线索档案。

移动端将线索档案改为可展开抽屉。

### NPC 对话视图

支持自由输入和快捷问题。快捷问题包括询问时间线、询问人物、询问物证、追问矛盾、询问规则。

AI 回复后，后端记录对话，并根据规则将新获得的线索写入已收集线索档案。

### 真相报告页

玩家填写：

- 事件真正经过。
- 关键人物或责任人。
- 关键物证。
- 怪谈规则的真实含义。
- 哪些 NPC 撒谎以及原因。
- 自己的最终结论。

报告必须引用玩家已收集线索，不能凭空填写所有答案。

### 结局复盘页

展示真相还原度、线索利用度、规则解释度、误判点、遗漏线索和最终结局。复盘文本由 DeepSeek V4 Pro 根据标准答案和玩家报告生成。

## 后端架构

后端采用 Spring Boot 分层架构，按领域拆包：

- `case`：案件模板、案件详情、案件规则。
- `game`：玩家会话、案件阶段、进度推进。
- `npc`：NPC 档案、知识边界、对话生成。
- `clue`：已收集线索、线索分类、标记状态。
- `report`：真相报告、评分、结局。
- `ai`：硅基流动 DeepSeek V4 Pro 适配层。
- `user`：游客玩家和后续用户体系预留。

核心服务：

- `CaseService`：案件列表、案件详情、地点、规则、初始任务。
- `GameSessionService`：开始案件、保存进度、阶段推进、判断解锁。
- `NpcDialogueService`：构造 AI prompt，调用模型，让 NPC 按知识边界回答。
- `ClueService`：把对话或探索中获得的线索写入已收集线索档案。
- `TruthReportService`：提交真相报告、对比标准答案、生成评分复盘和结局。
- `AiNarrativeClient`：统一 AI 接口，第一版实现为 `SiliconFlowDeepSeekClient`。

## 数据库设计

建议 MySQL 表：

- `users`：用户表，第一版创建游客用户。
- `cases`：案件主表。
- `case_rules`：怪谈规则。
- `case_locations`：案件地点。
- `case_npcs`：NPC 档案。
- `npc_knowledge`：NPC 已知事实、可透露线索、隐瞒内容。
- `case_clues`：案件标准线索池。
- `game_sessions`：玩家某次案件进度。
- `dialogue_messages`：玩家和 NPC 对话记录。
- `collected_clues`：玩家已收集线索档案。
- `truth_reports`：玩家提交的真相报告。
- `report_scores`：评分明细和结局。

关键规则：

- 案件真相、标准线索、NPC 知识边界保存在数据库，后端是唯一权威。
- AI 每次只能收到当前 NPC 可知道的信息，不能拿到完整真相。
- 玩家已收集线索单独落表，前端线索板只展示玩家实际获得过的信息。
- DeepSeek V4 Pro 用于语言表现、线索提取辅助和复盘，不用于决定事实真相。

## AI 调用设计

AI 只承担三类任务：

1. NPC 对话生成。
2. 线索提取辅助。
3. 真相报告复盘。

AI 不能：

- 修改案件真相。
- 新增决定性凶手或责任人。
- 泄露 NPC 不知道的信息。
- 绕过阶段限制提前透露后期线索。
- 把怪谈规则直接解释成最终答案。
- 生成与数据库标准线索冲突的事实。

NPC 对话调用输入包括：

- 当前案件标题。
- 当前阶段。
- 当前地点。
- NPC 名字、身份、性格、说话风格。
- NPC 已知事实。
- NPC 可透露线索。
- NPC 必须隐瞒内容。
- 玩家已收集线索摘要。
- 玩家最新问题。
- 输出格式要求。

建议 AI 输出 JSON：

```json
{
  "reply": "NPC 的自然语言回复",
  "mood": "nervous",
  "revealedClueCodes": ["CLOCK_MECHANISM_TRACE"],
  "ruleHints": ["第十三下钟声可能不是超自然现象"],
  "suspicionDelta": 5
}
```

后端校验：

- `revealedClueCodes` 必须存在于当前案件标准线索池。
- 当前阶段未解锁的线索不能入库。
- 如果 AI 回复泄露标准真相，后端丢弃并重试。
- 对话记录保存原始提问、AI 回复、触发线索和阶段信息。

评分输出建议：

```json
{
  "truthScore": 82,
  "clueScore": 75,
  "ruleScore": 60,
  "ending": "truth_partial",
  "summary": "你找到了镇长失踪的核心原因，但没有解释红手套和旧矿难的完整关系。",
  "missedPoints": ["红手套属于遇难者家属", "钟声是人为启动的机械信号"]
}
```

## 前端设计

整体风格为恐怖怪谈档案馆：

- 背景：深黑绿、雾灰、旧墙纸纹理。
- 强调色：暗红、蜡烛黄、锈铁色。
- 标题偏档案或打字机气质，正文保持易读。
- 动效使用轻微闪烁、雾层移动、档案纸展开、线索出现时短暂抖动。
- 避免大面积赛博霓虹、过度血腥和廉价跳吓。

Pinia store 拆分：

- `caseStore`：案件列表和详情。
- `sessionStore`：当前玩家会话、阶段、地点。
- `dialogueStore`：当前 NPC 对话记录。
- `clueStore`：已收集线索档案。
- `reportStore`：真相报告草稿和评分结果。

关键交互：

- 进入案件前先阅读规则，但不能判断真假。
- 点击地点切换 NPC 和地点描述。
- 对话输入框旁边提供快捷问题按钮。
- 新线索出现时自动写入线索档案，并高亮提示。
- 阶段推进时出现“调查阶段更新”。
- 违反规则时触发异常反馈，例如文字短暂错位、NPC 语气改变、地点锁定提示。

## 接口设计

案件相关：

```http
GET /api/cases
GET /api/cases/{caseId}
GET /api/cases/{caseId}/rules
GET /api/cases/{caseId}/locations
```

会话相关：

```http
POST /api/sessions
GET /api/sessions/{sessionId}
POST /api/sessions/{sessionId}/locations/{locationId}/visit
POST /api/sessions/{sessionId}/advance
```

NPC 对话：

```http
GET /api/sessions/{sessionId}/npcs
GET /api/sessions/{sessionId}/npcs/{npcId}/messages
POST /api/sessions/{sessionId}/npcs/{npcId}/messages
```

线索档案：

```http
GET /api/sessions/{sessionId}/clues
PATCH /api/sessions/{sessionId}/clues/{clueId}
```

真相报告：

```http
POST /api/sessions/{sessionId}/reports
GET /api/reports/{reportId}
```

游客玩家：

```http
POST /api/guest
GET /api/players/{playerId}/sessions
```

## 开发里程碑

### M1：项目骨架

- Spring Boot 后端项目。
- Vue 前端项目。
- MySQL 配置。
- 基础跨域、统一响应、异常处理。
- 游客玩家接口。

### M2：案件模板系统

- 建立案件、规则、地点、NPC、线索数据库表。
- 写入 3 个案件的种子数据。
- 完成案件列表、详情、规则、地点接口。

### M3：游戏会话与阶段推进

- 创建游客会话。
- 开始案件。
- 记录当前阶段、已访问地点、当前 NPC。
- 根据关键线索推进阶段。

### M4：NPC 对话与 AI 适配层

- 接入硅基流动 DeepSeek V4 Pro。
- 实现 `AiNarrativeClient`。
- 构造 NPC 对话 prompt。
- 保存对话记录。
- 校验 AI 返回的线索 code。

### M5：线索档案

- 收集线索入库。
- 前端展示已收集线索。
- 支持标记重要、可疑、已解释。
- 支持按人物、时间、地点、物证、规则、矛盾分类。

### M6：真相报告与评分

- 前端报告表单。
- 后端标准答案对比。
- AI 生成评分复盘。
- 生成多结局。

### M7：恐怖怪谈视觉打磨

- 案件档案馆 UI。
- 调查主界面三栏布局。
- 规则卡片与异常反馈。
- 移动端适配。
- 基础动效。

## MVP 完成标准

用户能以游客身份进入系统，选择 3 个案件之一，阅读规则，开始调查，和 NPC 对话，收集并查看线索，推进阶段，提交真相报告，获得 AI 评分和结局复盘。

## 非目标

第一版不做：

- 完整注册登录和权限系统。
- 玩家自定义案件生成。
- 多人协作调查。
- 复杂拖拽连线线索板。
- AI 自动改写案件真相。
- 语音对话。

这些能力可以作为后续版本规划。
